## Root Cause

- The Windows packaged server smoke was not actually closed:
  - the Windows zip extraction / cleanup path in `scripts/release/smoke-server-release-assets.mjs` still carried Windows-specific fragility around extraction, process teardown, and retryable cleanup
  - the packaged server bundle was launching `start-clawstudio-server.cmd`, which left a lingering `cmd.exe` parent and caused transient `EBUSY` cleanup failures
  - after switching to direct binary launch, the real packaged server still returned `/claw/health/ready = 503` because the default server control plane projected the bound server endpoint as `inactive`, so readiness could never pass
- The remaining release families were blocked by host-environment gaps rather than by the repaired server bundle logic:
  - desktop packaged-launch still fails when Node tries to `spawnSync` the NSIS installer or packaged desktop exe on this sandboxed Windows host with `EPERM`
  - linux container packaging on this host cannot produce a linux server binary because WSL access is denied, the `x86_64-unknown-linux-gnu` Rust target is not installed, and `x86_64-linux-gnu-gcc` is absent
  - kubernetes bundle packaging can emit artifacts, but live chart smoke remains host-dependent and this host does not expose `helm`

## Implemented Fix

- Hardened the Windows server bundle smoke path:
  - `scripts/release/smoke-server-release-assets.mjs`
    - aligned Windows zip extraction with the release packager flow
    - added retryable Windows directory cleanup
    - added Windows process-stop fallback logic
    - switched packaged Windows launch from `start-clawstudio-server.cmd` to the shipped `bin/sdkwork-clawstudio-server.exe`
    - normalized the recorded launcher path to `bin/sdkwork-clawstudio-server.exe`
  - `scripts/release/smoke-server-release-assets.test.mjs`
    - added and updated regression coverage for extraction, teardown fallback, retryable cleanup, and direct Windows binary launch
- Fixed the real packaged server readiness source instead of papering over the smoke:
  - `packages/sdkwork-clawstudio-server/src-host/src/http/routes/health.rs`
    - added a server-mode readiness fallback that treats the packaged server as ready when `claw-manage-http` is published with a reachable endpoint and `studio_public_api` is present
    - preserved the existing inactive manage/OpenClaw runtime and gateway projections by default instead of rewriting control-plane truth
  - `packages/sdkwork-clawstudio-server/src-host/src/main.rs`
    - preserved the negative `/claw/health/ready` case through an explicit inactive-provider test
    - added a positive default-server readiness test so the real server state must now return `200`
  - `packages/sdkwork-clawstudio-server/src-host/src/bootstrap.rs`
    - reverted the earlier broad control-plane projection change and kept the default `claw-manage-http` host publication semantics intact
- Carried forward the earlier Windows desktop packaging fixes that were already landed in this Step 11 loop:
