## Root Cause

- Step 09 had already produced the real runtime evidence for:
  - built-in OpenClaw startup readiness
  - packaged startup smoke
  - local AI proxy runtime evidence
  - Instance Detail consistency surfaces
- Step 10 was still open because that evidence was not fully frozen behind formal release-blocking gates:
  - several mandatory OpenClaw fact-source tests still relied on targeted/manual execution instead of parity runners
  - `check:automation` did not yet verify that those fact-source tests stayed wired into the official gate graph
  - release and deployment docs/tests still described a stale packaged deployment surface such as `deploy/Dockerfile` and `deploy/docker-compose.yml`, while the actual bundle layout uses `deploy/docker/*`
  - `scripts/prepare-openclaw-runtime.test.mjs` still depended on an external `tar` executable on Windows, which kept `check:desktop` environment-sensitive
  - `scripts/sdkwork-host-runtime-contract.test.ts` still forbade any `getDesktopKernelInfo()` read in desktop bootstrap even though Step 09 had explicitly approved reading `kernel.getInfo().localAiProxy` for startup evidence
