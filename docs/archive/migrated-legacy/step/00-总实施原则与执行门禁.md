# Step 00 - 总实施原则与执行门禁

## 1. Step Card

| 项 | 内容 |
| --- | --- |
| 执行模式 | 强串行 |
| 前置 | 无 |
| 主写入范围 | `docs/step/` `docs/review/` `docs/release/` `package.json` `scripts/check-*` |
| 执行输入 | `docs/架构/README`、`02`、`11`、`12`、`14`、`16`、`17`、`AGENTS.md`、根 `package.json` |
| 本步非目标 | 不改业务功能；不直接改聊天、实例、Provider、打包逻辑 |
| 最小输出 | 统一执行规则、质量门禁、证据口径、检查点命名、回写要求 |

## 2. 设计

- 采用“架构先行、证据先行、回归先行”的执行模型。
- 对托管 OpenClaw 的红线一次冻结：所有模型访问统一经 Local Proxy；Feature 不直拼供应商 API；Host 必须保持薄层。
- 所有 step 都必须以 package root 导出、服务层编排、平台契约调用为唯一工程入口。

## 3. 实施落地规划

1. 冻结 `00-13` 主步骤、波次、串并行规则和检查点命名。
2. 冻结统一证据目录：`docs/step/`、`docs/review/`、`docs/release/`。
3. 冻结统一门禁命令矩阵：`pnpm.cmd lint`、`check:arch`、`check:parity`、`check:desktop`、`check:server`、`check:release-flow` 的适用边界。
4. 冻结架构回写规则：每个 step 完成后必须回写受影响的 `docs/架构/*` 和 `docs/release/*`。
5. 冻结并行原则：先串行主脊柱，再按写入范围并行，不允许多车道主写同一高风险文件。

## 4. 测试计划

- 校对 `docs/step/README.md` 与 `00-13`、`90+` 是否完整。
- 验证门禁命令与当前 `package.json` 脚本一致。
- 验证每个 step 都具备设计、实施、测试、验证、检查点字段。

## 5. 结果验证

- 后续任一步都能明确回答“改哪里、怎么测、怎么验、失败如何回滚”。
- 所有执行人员都共享同一套口径，不再自行发明 step 模板。
- 发布前可以直接从 step 文档反推回归清单和证据目录。

## 6. 检查点

- `CP00-1`：`README + 00-13 + 90+` 主骨架齐备。
- `CP00-2`：门禁命令与适用边界冻结。
- `CP00-3`：架构回写、review、release 证据路径冻结。
- `CP00-4`：串并行规则冻结。

### 6.1 推荐并行车道

- `00-A`：step 模板与检查点规则
- `00-B`：命令矩阵与门禁口径
- `00-C`：review/release 证据模板
- 收口要求：仅 `00-Owner` 合并最终规则

### 6.2 完成后必须回写的架构文档

- `docs/架构/README.md`
- `docs/架构/02-架构标准与总体设计.md`

### 6.3 推荐 review 产物

- `docs/review/step-00-执行卡-YYYY-MM-DD.md`
- `docs/review/step-00-门禁冻结决议-YYYY-MM-DD.md`
- `docs/review/step-00-证据口径与验收规则-YYYY-MM-DD.md`

### 6.4 架构能力闭环判定

- 后续 `00-13` 全部 step 都必须共享同一套输入、输出、证据、回滚和验收口径。
- 如果只是写了原则，未形成门禁命令、模板和 review/release 证据规则，本 step 不算闭环。

### 6.5 快速完整执行建议

- 本步只做规则冻结，不扩散业务实现；推荐“规则车道、模板车道、验证口径车道”并行，当天统一收口。
- 任一步需要例外规则时，先回写本步文档和 `docs/review/` 决议，再继续编码。

## 7. 风险与回滚

- 风险：规则太松会失控，太严会拖慢推进。
- 回滚：优先调整本 step 文档，不在后续 step 中私自绕过规则。

## 8. 完成定义

- 形成统一模板、统一门禁、统一证据口径。
- 后续 step 不再需要补解释性流程文档。
